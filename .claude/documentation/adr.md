# ADR (Architecture Decision Record)

기술적 결정을 ADR로 기록합니다.

## 지시사항

1. **결정 내용 파악**: 현재 대화에서 논의된 기술적 결정을 분석합니다.
   - 어떤 문제를 해결하려 했는가?
   - 어떤 옵션들을 고려했는가?
   - 실제로 시도했다가 실패한 접근은 없는가?
   - 최종적으로 무엇을 선택했는가?

2. **ADR 번호 결정**: `docs/adr/` 폴더의 기존 파일을 확인하여 다음 번호를 부여합니다.
   - 예: 기존에 `0002_xxx.md`가 있으면 `0003_xxx.md`

3. **문서 생성**: `docs/adr/` 폴더에 아래 형식으로 문서를 생성합니다.
   - **파일명**: `NNNN_UPPER_SNAKE_CASE.md`
   - **예시**: `0003_EMOTION_OVER_TAILWIND.md`

4. **템플릿 형식**:

```markdown

tags: [태그1, 태그2]
---

|------|------|
| **상태** | 수락됨 |

| **결정자** | @[사용자명 또는 빈칸] |
| **관련 이슈** | [있으면 #번호] |

## 작성 원칙

| 원칙 | 설명 |
|------|------|
| **서사형 구조** | "문제 → 탐색 → 시행착오 → 선택 → 결과" 흐름으로 독자가 의사결정 과정을 따라갈 수 있게 |
| **시행착오 솔직하게** | 실패한 접근과 그 이유를 기록 — 같은 실수를 반복하지 않기 위해 가장 가치 있는 정보 |
| **Before/After 코드** | 실제 프로덕션 코드에 가까운 수준의 비교 예시 |
| **정량적 결과** | 파일 수, 빌드 시간, 번들 크기, 에러율 등 측정 가능한 수치 |
| **프로젝트 맥락** | 팀 규모, 기존 인프라, 서비스 특성 등 "왜 우리 프로젝트에서 이 선택이 맞았는가" |
| **간결함 유지** | 토큰 낭비 없이 핵심만 전달, 불필요한 반복 제거 |

## 사용 예시

기술적 결정을 내린 후 대화창에 `/documentation adr` 입력하면 자동으로 ADR이 생성됩니다.

### ADR이 필요한 상황
- 새 라이브러리 도입 결정
- 아키텍처 패턴 선택
- 기존 방식 대체/전환
- 성능 최적화 전략 결정
- 두 가지 이상 대안을 비교 후 하나를 선택한 경우
